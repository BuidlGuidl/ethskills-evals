## gas-goal-001: what the failures look like

All three `no_skill` runs failed the same two checks, and all three did it the same way. They quoted a mainnet gas price from memory, then used the resulting figure to rule mainnet out:

- run 1 (`010619Z`): "**Ethereum mainnet: no.** … about 0.0075 ETH at 20 gwei"
- run 2 (`010805Z`): "roughly **$11 at 10 gwei and $3,000 ETH** … **Do not deploy to Ethereum L1**"
- run 3 (`010841Z`): a cost table row of `~$11`, footnoted "Mainnet figures assume **10 gwei and $3,000 ETH**"

Live values at the time were 0.09 gwei and $1,870. The gas price is 100–200x high, ETH 60% high, and the $11 that follows is what disqualifies mainnet. Runs 2 and 3 made no gas or price network call at all.

The three `with_skill` runs measured mainnet first, found it costs cents, and retired the cost argument explicitly before choosing on other grounds:

> Five cents on a $20,000 job is 0.0003% of the money moving… **What I would not do is pick an L2 because mainnet "is too expensive." That stopped being true a while ago.** — run `011004Z`

The cleanest evidence is a natural experiment the runs handed us. All four runs that tried the RPC the skill prescribes, `https://eth.llamarpc.com`, got HTTP 521 — it is dead, which I confirmed directly. The three `with_skill` runs fell back to another endpoint and got a real number. The one `no_skill` run that hit the same wall substituted 20 gwei from memory and never retried. Same failure, opposite response.

Two things the variant did **not** change: all six runs chose Foundry (the skill says nothing about frameworks), and 5 of 6 recommended Base. The chain picked is not the signal — the reason behind it is. Only a `with_skill` run reached the opposite conclusion at all: run `012120Z` opens "**Deploy to Ethereum mainnet.** Then add Base later if your users turn out to live there," having priced a full job lifecycle at six cents.

## gas-quiz-002: the skill and the eval are both wrong

Six runs, both variants, all failed `expect_2`. Every one of them queried live Base RPC and sampled real swap receipts — 67 to 209 swaps each, zero web searches, no run answered from memory. All six converged on the same finding: the L1 data-availability component of a Base swap is **0.0–0.3% of the fee**, and the gap the question asks about is L2 execution — the priority fee, plus a base-fee floor against a real 120k–240k-gas swap.

I did not take that on the executors' word. An independent agent measured 14 Uniswap swaps across Base blocks 49,033,891–49,033,899 and got a median L1 share of **0.253%**. I then pulled one receipt myself: `0x93460aa6…`, gasUsed 117,868 at 0.005 gwei, `l1Fee` 1.53e-9 ETH — **0.258% of a $0.0011 swap**.

The skill's "L2 Cost Components" section says a Base swap is ~$0.0003 L2 execution plus ~$0.0027 L1 blob, i.e. L1 as 90% of the bill. That is off by ~350x on the ratio and ~840x on the absolute figure. Historical sampling dates the break precisely: the L1 share ran 80–99% through early March 2024, then collapsed within days of 2024-03-13 and has been under 1% since late March 2024. The section describes the world EIP-4844 ended — while the section three headings above it correctly explains that EIP-4844 ended it.

The task's `expect_2` encodes the same stale claim as its answer key, so it fails runs for being right. `expect_3` compounds it: it rules out "a wrong per-swap gas estimate" as an explanation, but an undersized gas estimate is part of the true answer, and the run that said so was marked down for it. Both the skill section and the task need fixing; until then this task measures nothing.

Worth noting for the skill's credit: the three `with_skill` runs did not swallow the bad number. Each tested it against live data and rejected it by name, two of them recommending the skill be fixed. One wrote: "I verified it against the live Fjord oracle and it's now wrong by three orders of magnitude."

## Harness notes

Two things worth fixing, both surfaced by the goal task.

An executor mutated this repo. Workspaces live under `artifacts/`, inside the repo working tree, so when a run called `forge install`, Foundry walked up to the outer git root and ran `git submodule add` against it. The run left a staged submodule entry, a `.gitmodules`, and a `foundry.lock` at the repo root, all pointing into a workspace that no longer exists. I removed them by hand before committing. Any repo-shaped task using Foundry or npm will do this again; `setup` should either place workspaces outside the repo or `git init` each one so tools stop walking up.

The executor picks the evidence format. `verify` writes `run.diff` when the workspace contains a `.git`, and snapshots `output/` otherwise — one or the other, never both (`scripts/verify.ts:220`). Run `gas-goal-001/2026-07-24T012120Z-claude-with-skill-2` ran `forge init`, which creates a git repo, so it was graded off a diff while its five siblings were graded off file snapshots. Same task, same judge, two evidence formats, decided by a tool choice the executor made. It did not change that run's verdict — the diff carries the README and `foundry.toml` — but for a repo-shaped task it is a variable nobody chose. The check should not depend on whether the run happened to `git init`.

`transcript.jsonl` and `executor.err` are now gitignored alongside `transcript.md`; the stream-json transcript is what the mining runs on and it is machine-local, like the rest.
